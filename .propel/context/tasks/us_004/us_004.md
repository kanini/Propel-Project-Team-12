# User Story - US_004

## Story ID

- ID: US_004

## Story Title

- Account Lockout After Failed Login Attempts

## Description

- As a security administrator, I want the system to lock user accounts after multiple failed login attempts, so that brute-force attacks are prevented and user accounts remain secure.

## Acceptance Criteria

1. **Given** a user has entered incorrect credentials 5 consecutive times, **When** they attempt a 6th login, **Then** the system locks the account and displays "Your account has been locked due to multiple failed login attempts. Please try again in 15 minutes or contact support."

2. **Given** a user has entered incorrect credentials 3 times, **When** they attempt a 4th login, **Then** the system displays a warning "2 attempts remaining before account lockout" to alert the user.

3. **Given** an account is locked, **When** the 15-minute lockout period expires, **Then** the account is automatically unlocked and the user can attempt to log in again with a fresh 5-attempt counter.

4. **Given** a user successfully logs in before reaching 5 failed attempts, **When** they authenticate successfully, **Then** the failed attempt counter resets to zero.

5. **Given** an account is locked, **When** the user attempts to log in during the lockout period, **Then** the system displays the remaining lockout time "Account locked. Try again in X minutes" without accepting credentials.

6. **Given** an account is locked, **When** clicking "Contact Support" or "Forgot Password", **Then** the user is directed to appropriate recovery options regardless of lockout status.

7. **Given** a lockout event occurs, **When** the account is locked, **Then** the system logs the event in the audit trail with timestamp, IP address, and user identifier for security review.

## Edge Cases

- What happens if a user forgets their password during lockout? (Password reset flow is available; successful reset unlocks account)
- How does the system handle lockout across multiple IP addresses? (Lockout is per-account, not per-IP, for stronger security)
- What happens if a Staff/Admin account is locked? (Same behavior; contact support option emphasized for operational continuity)
- How does the system handle distributed brute-force attacks? (Rate limiting at API gateway level provides additional protection - see EP-011)
