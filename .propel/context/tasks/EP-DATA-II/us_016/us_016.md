# User Story - US_016

## Story ID

* ID: US_016

## Story Title

* Insurance, No-Show History, and Audit Retention

## Description

* As a developer, I want InsuranceRecord and NoShowHistory entities created plus HIPAA-compliant 7-year audit log retention policy, so that insurance validation, no-show risk, and compliance retention are supported.

## Acceptance Criteria

1. **Given** InsuranceRecord entity is defined, **When** I inspect the class, **Then** it contains ID, provider name, accepted ID patterns (regex), coverage type (enum), active status, and validation rules — this is reference data used for FR-021 insurance pre-check.

2. **Given** NoShowHistory entity is defined, **When** I inspect the class, **Then** it contains ID, patient reference (FK), total appointment count, no-show count, confirmation response rate, average lead time for no-shows, last calculated risk score, and last updated timestamp.

3. **Given** audit retention is 7 years (DR-007), **When** I inspect the database configuration, **Then** a scheduled job or database policy exists that prevents deletion of audit logs younger than 7 years. Logs older than 7 years may be archived.

4. **Given** the migration is applied, **When** I verify the schema, **Then** InsuranceRecord has an index on (provider_name, active) and NoShowHistory has a unique constraint on patient_id.

## Edge Cases

* What happens when audit logs exceed storage limits? Archival strategy should compress and move records older than 7 years to cold storage.
* How does the system handle insurance records with invalid regex patterns? Application-level validation tests regex patterns on save to prevent invalid patterns from being stored.
