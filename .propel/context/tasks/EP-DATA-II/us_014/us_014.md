# User Story - US_014

## Story ID

* ID: US_014

## Story Title

* Waitlist and Intake Record Entities

## Description

* As a developer, I want WaitlistEntry and IntakeRecord entities with proper relationships and constraints, so that patient waitlist preferences and pre-visit intake data are persisted correctly.

## Acceptance Criteria

1. **Given** WaitlistEntry entity is defined, **When** I inspect the class, **Then** it contains ID, patient reference (FK), provider reference (FK), preferred start date, preferred end date, preferred time range, notification preference (enum: SMS/Email/Both), priority timestamp, and status.

2. **Given** IntakeRecord entity is defined, **When** I inspect the class, **Then** it contains ID, appointment reference (FK), patient reference (FK), intake mode (enum: AI/Manual), medical history (JSONB), current medications (JSONB), allergies (JSONB), visit concerns, insurance validation status, insurance record reference (FK nullable), and completion status.

3. **Given** the migration is applied, **When** I query the schema, **Then** foreign keys exist from WaitlistEntry to User and Provider, and from IntakeRecord to Appointment and User, with proper indexes on patient reference.

4. **Given** JSONB columns are used, **When** structured health data is stored, **Then** PostgreSQL JSONB type enables efficient querying and indexing of nested intake data.

## Edge Cases

* What happens when an appointment is cancelled but has an intake record? IntakeRecord should remain for historical reference with soft delete flag.
* How does the system handle duplicate waitlist entries for the same patient/provider/date? Application-level validation prevents duplicates; unique constraint on (patient_id, provider_id, preferred_start_date).
