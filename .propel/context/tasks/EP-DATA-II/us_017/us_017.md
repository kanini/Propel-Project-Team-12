# User Story - US_017

## Story ID

* ID: US_017

## Story Title

* Reference Data Seeders

## Description

* As a developer, I want data seeders for insurance provider reference data and provider availability schedules, so that the system has realistic test data for insurance pre-check validation and appointment booking.

## Acceptance Criteria

1. **Given** insurance data seeder is implemented, **When** I run the seeder, **Then** at least 10 dummy insurance provider records are created with varied provider names, ID patterns, and coverage types.

2. **Given** provider data seeder is implemented, **When** I run the seeder, **Then** at least 5 providers are created with different specialties and weekly availability schedules generating time slots for the next 30 days.

3. **Given** seeders are idempotent, **When** I run the seeders multiple times, **Then** existing records are not duplicated — seeders check for existing data before inserting.

4. **Given** the development environment is initialized, **When** I run `dotnet ef database update` followed by the seeder command, **Then** both insurance and provider data are populated and queryable.

## Edge Cases

* What happens when the seeder runs against a production database? Seeders should be environment-guarded to execute only in Development/Staging environments.
* How does the system handle seeder failures mid-execution? Seeder should run in a transaction to ensure all-or-nothing data population.
