# User Story - US_015

## Story ID

* ID: US_015

## Story Title

* Medical Code and Notification Entities

## Description

* As a developer, I want MedicalCode and Notification entities with confidence scoring, delivery tracking, and proper relationships, so that AI code suggestions and multi-channel notification history are persisted.

## Acceptance Criteria

1. **Given** MedicalCode entity is defined, **When** I inspect the class, **Then** it contains ID, extracted data reference (FK), code system (enum: ICD10/CPT), code value, description, confidence score (0-100), verification status (enum: AISuggested/Accepted/Modified/Rejected), and verifier reference (FK nullable).

2. **Given** Notification entity is defined, **When** I inspect the class, **Then** it contains ID, recipient reference (FK), appointment reference (FK nullable), channel type (enum: SMS/Email), template name, status (enum: Pending/Sent/Failed/Delivered), scheduled time, sent time, delivery confirmation, retry count, and last error message.

3. **Given** migrations are applied, **When** I query the schema, **Then** indexes exist on Notification (recipient_id, scheduled_time, status) for efficient query of pending notifications and on MedicalCode (extracted_data_id) for retrieval by source data.

## Edge Cases

* What happens when a notification channel is not configured for a recipient? Notification should be created with status Failed and error message indicating missing channel configuration.
* How does the system handle medical codes with identical code values but different systems? Composite uniqueness on (extracted_data_id, code_system, code_value) prevents duplicates per extraction.
