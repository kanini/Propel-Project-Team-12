# User Story - US_019

## Story ID
* ID: US_019

## Story Title
* Database Indexes and Constraints

## Description
* As a database administrator, I want comprehensive indexes (B-tree, GIN, GiST) and constraints (foreign keys, unique, check) applied to all entities, so that query performance meets SLA requirements and data integrity is enforced at the database level.

## Acceptance Criteria
1. **Given** all entity models are defined, **When** I generate the database migration for indexes, **Then** I have:
   - B-tree indexes on all foreign key columns (PatientID, AppointmentID, DocumentID, etc.)
   - GIN index on JSONB columns for full-text search in MedicalHistory and Allergies
   - GiST index on pgvector embedding columns for cosine similarity queries
   - Composite unique indexes on business logic constraints (e.g., Appointment: SlotDateTime + ProviderID)

2. **Given** indexes are created, **When** I run EXPLAIN ANALYZE on common queries, **Then** index usage is confirmed:
   - Patient lookup by email uses `IX_Patients_Email_Unique` index (Index Scan, not Seq Scan)
   - ExtractedData low-confidence query uses `IX_ExtractedData_ConfidenceScore` filtered index
   - pgvector similarity search uses GiST index with `<=>` operator

3. **Given** constraints are defined, **When** I attempt constraint violations, **Then** the database rejects:
   - Duplicate email insertion: `ERROR: duplicate key value violates unique constraint "IX_Patients_Email_Unique"`
   - Invalid ConfidenceScore (e.g., 1.5): `ERROR: new row violates check constraint "CK_ExtractedData_ConfidenceScore"`
   - Foreign key violation (orphaned ExtractedData): `ERROR: insert or update violates foreign key constraint "FK_ExtractedData_DocumentID"`

4. **Given** the migration script is reviewed, **When** I check for performance impact, **Then** I confirm:
   - Non-blocking index creation using `CREATE INDEX CONCURRENTLY` for production deployments
   - Index build time estimated for 10K records baseline (< 30 seconds per index)

## Edge Cases
* What happens when GiST index build fails on large embedding datasets? (Retry with increased memory settings; fallback to sequential scan with performance degradation warning)
* How does system handle partial indexes on filtered columns (e.g., low-confidence only)? (Document filter expressions in migration; validate query planner uses partial index)

## Technical Notes
- Migration for indexes and constraints:
  ```csharp
  migrationBuilder.CreateIndex(
      name: "IX_Patients_Email_Unique",
      table: "Patients",
      column: "Email",
      unique: true);

  migrationBuilder.CreateIndex(
      name: "IX_Appointments_SlotDateTime_ProviderID_Unique",
      table: "Appointments",
      columns: new[] { "SlotDateTime", "ProviderID" },
      unique: true);

  migrationBuilder.CreateIndex(
      name: "IX_ExtractedData_ConfidenceScore",
      table: "ExtractedData",
      column: "ConfidenceScore",
      filter: "ConfidenceScore < 0.7"); // Partial index

  // GIN index for JSON full-text search
  migrationBuilder.Sql(@"
      CREATE INDEX IX_IntakeForms_MedicalHistory_GIN 
      ON IntakeForms USING GIN (to_tsvector('english', MedicalHistory));
  ");

  // GiST index for pgvector cosine similarity
  migrationBuilder.Sql(@"
      CREATE INDEX IX_ClinicalDocuments_EmbeddingVector_GiST 
      ON ClinicalDocuments USING gist (EmbeddingVector vector_cosine_ops);
  ");

  // Check constraint
  migrationBuilder.AddCheckConstraint(
      name: "CK_ExtractedData_ConfidenceScore",
      table: "ExtractedData",
      sql: "ConfidenceScore >= 0.0 AND ConfidenceScore <= 1.0");

  // Foreign key with cascade
  migrationBuilder.AddForeignKey(
      name: "FK_Appointments_Patients_PatientID",
      table: "Appointments",
      column: "PatientID",
      principalTable: "Patients",
      principalColumn: "PatientID",
      onDelete: ReferentialAction.Restrict); // Or Cascade based on design
  ```
- Query plan validation:
  ```sql
  EXPLAIN ANALYZE 
  SELECT * FROM Patients WHERE Email = 'user@example.com';
  -- Expected: Index Scan using IX_Patients_Email_Unique

  EXPLAIN ANALYZE
  SELECT * FROM ClinicalDocuments 
  ORDER BY EmbeddingVector <=> '[0.1, 0.2, ...]'::vector LIMIT 5;
  -- Expected: Index Scan using IX_ClinicalDocuments_EmbeddingVector_GiST
  ```
- Index maintenance:
  - Regularly run `VACUUM ANALYZE` to update statistics
  - Monitor index bloat with `pg_stat_user_indexes`
  - Rebuild indexes quarterly if fragmentation > 30%
