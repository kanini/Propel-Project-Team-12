# User Story - US_024

## Story ID
* ID: US_024

## Story Title
* Database Backup Configuration

## Description
* As a DevOps engineer, I want automated daily database backups with 7-day point-in-time recovery enabled on Supabase free tier, so that the system can recover from data corruption or accidental deletion within defined RTO/RPO objectives (RTO ≤4 hours, RPO ≤24 hours).

## Acceptance Criteria
1. **Given** Supabase PostgreSQL is configured, **When** I enable point-in-time recovery, **Then** I confirm:
   - Supabase dashboard shows "Point-in-Time Recovery (PITR)" enabled
   - Retention period: 7 days (free tier limit)
   - Automated backups run daily at 02:00 UTC

2. **Given** backups are operational, **When** I review backup history in Supabase dashboard, **Then** I see:
   - List of daily backup timestamps for past 7 days
   - Backup size metrics (incremental changes only)
   - Status: Completed/Failed with error details if any

3. **Given** a data loss scenario occurs, **When** I initiate recovery, **Then** I can:
   - Select recovery point within 7-day window
   - Restore database to exact timestamp (e.g., "Restore to 2026-03-18 14:30:00 UTC")
   - Verification: Restored data matches pre-loss state (tested quarterly per acceptance criteria #4)

4. **Given** backup restoration must be tested, **When** I perform quarterly restoration drill, **Then** I:
   - Create test Supabase project from backup
   - Verify data integrity: row counts, foreign key constraints, index presence
   - Validate application connectivity to restored database
   - Document restoration time (must meet RTO ≤4 hours)
   - Record findings in `BACKUP_RESTORATION_TEST_LOG.md`

## Edge Cases
* What happens when restoration exceeds RTO of 4 hours? (Escalate to Supabase support; consider manual CSV export/import as fallback; document incident)
* How does system handle backups during high-write periods? (Supabase PITR uses Write-Ahead Log (WAL) replication, minimal performance impact; monitor replication lag)

## Technical Notes
- **Enabling PITR on Supabase:**
  1. Navigate to Supabase project dashboard
  2. Go to Database → Backups
  3. Enable "Point-in-Time Recovery" toggle
  4. Confirm 7-day retention (free tier default)
  5. Automated daily backups start immediately

- **Backup verification script:**
  ```bash
  # Check backup status via Supabase CLI
  supabase db backups list

  # Expected output:
  # Backup ID         | Created At           | Status
  # -----------------------------------------------------
  # backup-2026-03-19 | 2026-03-19 02:00:00  | Completed
  # backup-2026-03-18 | 2026-03-18 02:00:00  | Completed
  ```

- **Restoration procedure (quarterly drill):**
  ```markdown

  **Tester:** DevOps Engineer Name
  **Objective:** Verify 7-day PITR functionality and meet RTO ≤4 hours

  1. Create test Supabase project: `test-patient-access-restore`
  2. Navigate to Database → Backups → Restore
  3. Select recovery timestamp: 2026-03-18 14:00:00 UTC (24 hours ago)
  ```

- **Blob storage backup (separate from database):**
  - Supabase Storage automatically replicates to geographically separate region (meets DR-011)
  - Verify replication: Check Supabase dashboard → Storage → Settings → "Geo-replication enabled"
  - Manual verification: Upload test file, delete from primary, verify recovery from replica

- **Backup monitoring automation:**
  ```csharp
  // Background job (Hangfire) to verify daily backups
  [AutomaticRetry(Attempts = 0)]
  public async Task VerifyDailyBackup()
  {
      // Call Supabase Management API to check backup status
      var backupStatus = await _supabaseClient.GetLatestBackupStatus();

      if (backupStatus.Status != "Completed" || backupStatus.CreatedAt < DateTime.UtcNow.AddDays(-1))
      {
          // Alert: Backup failed or missing
          await _alertingService.SendAlert("Database backup failure detected");
      }
  }
  ```

- **RTO/RPO targets:**
  - **RTO (Recovery Time Objective)**: ≤4 hours (time to restore service)
  - **RPO (Recovery Point Objective)**: ≤24 hours (acceptable data loss window)
  - Free-tier constraints: 7-day PITR window meets RPO; restoration speed meets RTO

- **Backup documentation:**
  - Create `DISASTER_RECOVERY_PLAN.md` with:
    - Backup schedule (daily at 02:00 UTC)
    - Restoration procedures (step-by-step guide)
    - Contact list (Supabase support, on-call engineer)
    - Escalation matrix (if RTO exceeded)
