# User Story - US_015

## Story ID
* ID: US_015

## Story Title
* Appointment Entity Models with Optimistic Locking

## Description
* As a backend developer, I want Appointment and AppointmentSlot entities with RowVersion optimistic concurrency control, so that concurrent booking attempts on the same slot are detected and prevented through conflict resolution.

## Acceptance Criteria
1. **Given** the Domain layer exists, **When** I create entity classes, **Then** I have:
   - `Appointment.cs` with properties: AppointmentID (GUID), PatientID (FK), ProviderID (FK), SlotDateTime, Duration (minutes), Status (enum), BookingSource (enum), PreferredSlotSwapRequestID (FK nullable), RowVersion (byte array for optimistic locking), BookedAt, ArrivedAt (nullable), QueuePosition (int nullable)
   - `AppointmentSlot.cs` with properties: SlotID (GUID), ProviderID (FK), StartDateTime, EndDateTime, IsAvailable (bool), SlotType (enum: Regular/WalkIn/Emergency)
   - `PreferredSlotSwapRequest.cs` with properties: RequestID (GUID), AppointmentID (FK), PreferredSlotDateTime, RequestedAt, ProcessedAt (nullable), Status (enum: Pending/Swapped/Expired)

2. **Given** the entities are configured, **When** I use Fluent API to set up optimistic concurrency, **Then** `Appointment` entity has:
   - RowVersion column: `.Property(e => e.RowVersion).IsRowVersion()`
   - Concurrency token configured for EF Core to check on UPDATE
   - Foreign key to Patient: `.HasOne(a => a.Patient).WithMany(p => p.Appointments).HasForeignKey(a => a.PatientID)`

3. **Given** two concurrent booking requests target the same slot, **When** both transactions attempt to book the slot, **Then**:
   - First transaction commits successfully, marking slot as unavailable
   - Second transaction throws `DbUpdateConcurrencyException` due to RowVersion mismatch
   - API returns HTTP 409 Conflict with message: "Slot no longer available, please select another time"

4. **Given** the migration is created, **When** I review the SQL, **Then** I see:
   - `RowVersion` column as `BYTEA` (PostgreSQL) or `timestamp` (SQL Server) with automatic update trigger
   - Unique constraint on `(SlotDateTime, ProviderID)` for Appointment to prevent double-booking
   - Check constraint on AppointmentSlot ensuring `EndDateTime > StartDateTime`

## Edge Cases
* What happens when RowVersion is manually modified by external tool? (RowVersion is auto-managed by database; manual changes invalid and detected on next update)
* How does system handle orphaned PreferredSlotSwapRequests when appointments are cancelled? (Cascade delete via foreign key or background cleanup job marks expired after 48 hours)

## Technical Notes
- Appointment entity with optimistic locking:
  ```csharp
  public class Appointment
  {
      public Guid AppointmentID { get; set; }
      public Guid PatientID { get; set; }
      public Guid ProviderID { get; set; }
      public DateTime SlotDateTime { get; set; }
      public int Duration { get; set; } // Minutes
      public AppointmentStatus Status { get; set; }
      public BookingSource BookingSource { get; set; }
      public Guid? PreferredSlotSwapRequestID { get; set; }
      public byte[] RowVersion { get; set; } = Array.Empty<byte>(); // Optimistic locking
      public DateTime BookedAt { get; set; }
      public DateTime? ArrivedAt { get; set; }
      public int? QueuePosition { get; set; }
      
      // Navigation properties
      public Patient Patient { get; set; } = null!;
      public PreferredSlotSwapRequest? PreferredSlotSwapRequest { get; set; }
  }

  public enum AppointmentStatus { Scheduled, Arrived, Completed, Cancelled, NoShow }
  public enum BookingSource { Online, WalkIn, Staff }
  ```
- Fluent API for optimistic concurrency:
  ```csharp
  modelBuilder.Entity<Appointment>(entity =>
  {
      entity.HasKey(e => e.AppointmentID);
      entity.Property(e => e.RowVersion).IsRowVersion(); // Optimistic locking
      entity.HasIndex(e => new { e.SlotDateTime, e.ProviderID }).IsUnique();
      entity.HasOne(e => e.Patient)
            .WithMany(p => p.Appointments)
            .HasForeignKey(e => e.PatientID)
            .OnDelete(DeleteBehavior.Restrict); // Prevent cascade
  });
  ```
- Handling concurrency exceptions in service layer:
  ```csharp
  try
  {
      await _context.SaveChangesAsync();
  }
  catch (DbUpdateConcurrencyException ex)
  {
      // Log and return 409 Conflict
      _logger.LogWarning($"Concurrent booking conflict: {ex.Message}");
      throw new ConcurrencyConflictException("Slot no longer available");
  }
  ```
