# User Story - US_021

## Story ID
* ID: US_021

## Story Title
* Immutable AuditLog Database Trigger

## Description
* As a database administrator, I want a PostgreSQL trigger that prevents UPDATE and DELETE operations on the AuditLogs table, so that the audit trail remains immutable and complies with HIPAA recordkeeping requirements.

## Acceptance Criteria
1. **Given** the AuditLogs table exists, **When** I create a database trigger, **Then** it includes:
   - Trigger name: `TR_AuditLogs_Prevent_Modifications`
   - Trigger timing: BEFORE UPDATE OR DELETE
   - Trigger function: Raises exception with message "Audit logs are immutable. UPDATE and DELETE operations are not permitted."

2. **Given** the trigger is installed, **When** I attempt `UPDATE AuditLogs SET Action = 'Modified' WHERE AuditLogID = @id`, **Then** the database:
   - Raises exception: `ERROR: Audit logs are immutable`
   - Transaction rolls back
   - No rows are modified

3. **Given** the trigger is installed, **When** I attempt `DELETE FROM AuditLogs WHERE Timestamp < @cutoffDate`, **Then** the database:
   - Raises exception: `ERROR: Audit logs are immutable`
   - No rows are deleted
   - Archive process must use table partitioning instead

4. **Given** INSERT operations are still allowed, **When** I execute `INSERT INTO AuditLogs (...) VALUES (...)`, **Then** the operation succeeds (trigger only blocks UPDATE/DELETE).

## Edge Cases
* What happens when database administrator needs to correct erroneous audit log entry? (Manual correction prohibited; compensating entry (new INSERT) added with correction details; original entry preserved)
* How does system handle archival of old audit logs? (Use table partitioning by year; detach old partitions instead of DELETE; documented in US_024)

## Technical Notes
- PostgreSQL trigger function:
  ```sql
  CREATE OR REPLACE FUNCTION prevent_audit_log_modifications()
  RETURNS TRIGGER AS $$
  BEGIN
      RAISE EXCEPTION 'Audit logs are immutable. UPDATE and DELETE operations are not permitted.';
      RETURN NULL;
  END;
  $$ LANGUAGE plpgsql;
  ```
- Trigger creation:
  ```sql
  CREATE TRIGGER TR_AuditLogs_Prevent_Modifications
  BEFORE UPDATE OR DELETE ON "AuditLogs"
  FOR EACH ROW
  EXECUTE FUNCTION prevent_audit_log_modifications();
  ```
- EF Core migration:
  ```csharp
  migrationBuilder.Sql(@"
      CREATE OR REPLACE FUNCTION prevent_audit_log_modifications()
      RETURNS TRIGGER AS $$
      BEGIN
          RAISE EXCEPTION 'Audit logs are immutable. UPDATE and DELETE operations are not permitted.';
          RETURN NULL;
      END;
      $$ LANGUAGE plpgsql;

      CREATE TRIGGER TR_AuditLogs_Prevent_Modifications
      BEFORE UPDATE OR DELETE ON ""AuditLogs""
      FOR EACH ROW
      EXECUTE FUNCTION prevent_audit_log_modifications();
  ");
  ```
- Testing the trigger:
  ```sql
  -- Test INSERT (should succeed)
  INSERT INTO "AuditLogs" (AuditLogID, UserID, UserRole, Action, EntityType, EntityID, Timestamp, IPAddress, UserAgent, ChangeDetails)
  VALUES (gen_random_uuid(), gen_random_uuid(), 'Patient', 'Create', 'Appointment', gen_random_uuid(), NOW(), '192.168.1.1', 'TestAgent', '{}');

  -- Test UPDATE (should fail)
  UPDATE "AuditLogs" SET Action = 'Modified' WHERE AuditLogID = (SELECT AuditLogID FROM "AuditLogs" LIMIT 1);
  -- ERROR: Audit logs are immutable

  -- Test DELETE (should fail)
  DELETE FROM "AuditLogs" WHERE Timestamp < NOW() - INTERVAL '1 day';
  -- ERROR: Audit logs are immutable
  ```
- Archival strategy (documented, not implemented in this story):
  - Use table partitioning by year: `CREATE TABLE AuditLogs_2026 PARTITION OF AuditLogs FOR VALUES FROM ('2026-01-01') TO ('2027-01-01')`
  - Detach old partitions: `ALTER TABLE AuditLogs DETACH PARTITION AuditLogs_2019`
  - Move detached partition to cold storage or archive database
