# User Story - US_061

## Story ID

* ID: US_061

## Story Title

* Form Accessibility

## Description

* As a Patient with accessibility needs, I want all forms to have proper labels, error associations, and input assistance, so that I can fill out registration, booking, and intake forms without barriers.

## Acceptance Criteria

1. **Given** form accessibility (UXR-203), **When** a form renders, **Then** every input has a visible `<label>` element with a matching `for` attribute, and group labels use `<fieldset>` and `<legend>`.

2. **Given** inline validation, **When** a field fails validation, **Then** the error message is programmatically associated via `aria-describedby`, prefixed with an error icon, and announced by screen readers.

3. **Given** required fields, **When** a form displays, **Then** required fields show both a visual asterisk indicator and `aria-required="true"` attribute.

4. **Given** autocomplete, **When** personal data fields render (name, email, phone, address), **Then** appropriate `autocomplete` attribute values are set per HTML5 specification to aid autofill.

## Edge Cases

* What happens with complex multi-select fields (e.g., selecting specialties)? The component follows WAI-ARIA listbox pattern with `aria-multiselectable="true"` and announces selection count changes.
* How are date picker fields handled? Date pickers provide both keyboard-operable calendar widgets and a manual text input fallback accepting standard date formats.
