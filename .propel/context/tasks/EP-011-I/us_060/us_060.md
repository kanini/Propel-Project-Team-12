# User Story - US_060

## Story ID

* ID: US_060

## Story Title

* Keyboard Navigation & Focus Management

## Description

* As a Patient who relies on keyboard-only navigation, I want all interactive elements to be fully operable via keyboard with logical tab order and proper focus management, so that I can complete every task without a mouse.

## Acceptance Criteria

1. **Given** keyboard navigation (UXR-202), **When** I press Tab, **Then** focus moves in a logical reading order through all interactive elements (links, buttons, inputs, selects) and skip-to-content link is the first focusable element.

2. **Given** modal/dialog focus management (UXR-205), **When** a modal opens, **Then** focus is trapped within the modal, Escape closes it, and focus returns to the trigger element on close.

3. **Given** dropdown menus and custom components, **When** I interact via keyboard, **Then** Arrow keys navigate options, Enter/Space activates, and Escape dismisses, following WAI-ARIA combobox/listbox patterns.

4. **Given** multi-step workflows (booking, intake), **When** I advance to the next step, **Then** focus moves to the first interactive element of the new step with an ARIA live announcement of the step title.

## Edge Cases

* What happens when focus is on a disabled element? Disabled elements are removed from the tab order; visual indication and `aria-disabled="true"` explain why the element is unavailable.
* How does the system handle dynamically inserted content (e.g., validation errors)? Focus shifts to the first error message and an ARIA alert announces the error count.
