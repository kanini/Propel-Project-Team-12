# User Story - US_012

## Story ID

* ID: US_012

## Story Title

* Audit Log Entity and Referential Integrity

## Description

* As a developer, I want an immutable AuditLog entity with database triggers preventing UPDATE/DELETE and referential integrity constraints across all patient-related entities, so that compliance and data integrity are enforced at the database level.

## Acceptance Criteria

1. **Given** the AuditLog entity is defined, **When** I inspect the class, **Then** it contains ID, user reference (FK), timestamp, action type, resource type, resource ID, action details (JSONB), and IP address.

2. **Given** immutability is required (AD-007), **When** a migration is applied, **Then** database triggers exist on the `AuditLogs` table that raise exceptions on UPDATE or DELETE operations.

3. **Given** referential integrity is specified (DR-006), **When** I inspect the database schema, **Then** foreign key constraints link appointments, documents, intake records, and waitlist entries back to the patient User record.

4. **Given** the audit log is append-only, **When** I attempt to update or delete an audit record via EF Core, **Then** the operation fails with a database-level exception.

## Edge Cases

* What happens when audit log insertion fails? The originating transaction should still complete, with the audit failure logged to application error logs for investigation.
* How does the system handle high-volume audit writes? Audit log inserts should be batched or use async write patterns to avoid blocking main operations.
