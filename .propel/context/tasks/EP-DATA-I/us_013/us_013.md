# User Story - US_013

## Story ID

* ID: US_013

## Story Title

* Database Backup and Migration Infrastructure

## Description

* As a developer, I want database backup with 15-minute point-in-time recovery configured on Supabase and zero-downtime versioned migration scripts, so that data recovery and schema evolution are reliable and non-disruptive.

## Acceptance Criteria

1. **Given** Supabase manages backups, **When** I verify the database configuration, **Then** point-in-time recovery is enabled with a recovery window of at least 15-minute granularity.

2. **Given** versioned migrations are required, **When** I run `dotnet ef migrations list`, **Then** all migrations are listed in chronological order with timestamps and descriptive names.

3. **Given** zero-downtime is required, **When** a migration adds a new column, **Then** it uses a non-breaking schema change pattern (nullable column first, backfill, then constraint) documented in migration comments.

4. **Given** migration rollback is needed, **When** I run `dotnet ef database update <previous_migration>`, **Then** the schema reverts cleanly to the target migration state.

## Edge Cases

* What happens when a migration fails mid-execution? Migration should run in a transaction so partial changes are rolled back automatically.
* How does the system handle data loss from a failed backup? Supabase PITR allows recovery to any 15-minute checkpoint within the retention window.
