# User Story - US_011

## Story ID

* ID: US_011

## Story Title

* Clinical Document and Extracted Data Entities

## Description

* As a developer, I want ClinicalDocument and ExtractedClinicalData entities with processing status, confidence scores, and verification tracking, so that uploaded documents and their AI-extracted data points are properly persisted.

## Acceptance Criteria

1. **Given** the ClinicalDocument entity is defined, **When** I inspect the class, **Then** it contains ID, patient reference (FK), file name, file size, MIME type, storage path, processing status (enum: Uploaded/Processing/Completed/Failed), upload timestamp, and processing timestamp.

2. **Given** the ExtractedClinicalData entity is defined, **When** I inspect the class, **Then** it contains ID, source document reference (FK), patient reference (FK), data type (enum: Vital/Medication/Allergy/Diagnosis/LabResult), key, value, confidence score (0-100), verification status (enum: AISuggested/StaffVerified/Rejected), source page number, source text excerpt, and verifier reference.

3. **Given** the migration is applied, **When** I query the schema, **Then** foreign keys from ExtractedClinicalData to ClinicalDocument and to User (patient) exist with indexes on patient reference and data type for query performance.

4. **Given** pgvector support is configured, **When** I inspect the ExtractedClinicalData table, **Then** an optional vector column (`embedding vector(1536)`) exists for semantic similarity search.

## Edge Cases

* What happens when a clinical document is deleted? Extracted data should cascade delete since it's meaningless without the source.
* How does the system handle extremely large documents? File size validation at application level prevents persisting metadata for oversized files.
