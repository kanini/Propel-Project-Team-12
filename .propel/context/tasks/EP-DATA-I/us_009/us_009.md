# User Story - US_009

## Story ID

* ID: US_009

## Story Title

* User Entity and Migration

## Description

* As a developer, I want the User entity created with email uniqueness, credential hash, role, and status fields with EF Core configuration, so that user data can be persisted and queried securely.

## Acceptance Criteria

1. **Given** the User entity is defined, **When** I inspect the entity class, **Then** it contains fields for ID (GUID), email (unique), name, date of birth, phone, password hash, role (enum), status (enum), and timestamps (created, updated).

2. **Given** email uniqueness is required, **When** a migration is applied, **Then** a unique index exists on the `Email` column enforced at the database level.

3. **Given** EF Core configuration is implemented, **When** I inspect `UserConfiguration.cs`, **Then** column types, max lengths, required fields, and index definitions are configured via Fluent API.

4. **Given** the migration is applied, **When** I run `dotnet ef database update`, **Then** the `Users` table is created in PostgreSQL matching the entity definition with all constraints.

## Edge Cases

* What happens when a duplicate email insert is attempted? Database raises unique constraint violation caught as a descriptive application-level error.
* How does the system handle null or empty email values? Database NOT NULL constraint and application-level validation reject them before persistence.
