# User Story - US_010

## Story ID

* ID: US_010

## Story Title

* Appointment Entity and Migration

## Description

* As a developer, I want the Appointment entity with status lifecycle, provider reference, visit reason, and related TimeSlot and Provider entities, so that scheduling data is properly structured and queryable.

## Acceptance Criteria

1. **Given** the Appointment entity is defined, **When** I inspect the class, **Then** it contains ID, patient reference (FK), provider reference (FK), scheduled datetime, status (enum: Scheduled/Confirmed/Arrived/Completed/Cancelled/NoShow), visit reason, walk-in flag, preferred swap reference, no-show risk score, and cancellation notice hours.

2. **Given** the Provider entity is defined, **When** I inspect the class, **Then** it contains ID, name, specialty, and availability schedule with no login capability (reference entity only).

3. **Given** the TimeSlot entity is defined, **When** I inspect the class, **Then** it contains ID, provider reference (FK), start time, end time, and booking status with proper indexing for availability queries.

4. **Given** the migration is applied, **When** I query the database schema, **Then** foreign keys linking Appointment to User (patient) and Provider exist with proper cascade rules.

## Edge Cases

* What happens when an appointment references a deleted provider? Referential integrity should prevent provider deletion when appointments exist (RESTRICT).
* How does the system handle concurrent time slot bookings? Optimistic concurrency tokens should be defined on the TimeSlot entity.
