# User Story - US_019

## Story ID

* ID: US_019

## Story Title

* User Login and Session Management

## Description

* As a registered user, I want to log in with my email and password and receive a secure session with 15-minute auto-timeout, so that I can access my role-appropriate dashboard securely.

## Acceptance Criteria

1. **Given** I am on the login page, **When** I enter valid credentials, **Then** the system authenticates me, generates a JWT session token, stores it in Redis with 15-minute TTL, and redirects me to my role-appropriate dashboard.

2. **Given** I am logged in, **When** 15 minutes of inactivity pass, **Then** the session token expires in Redis, the frontend detects expiration, and I am redirected to the login page.

3. **Given** I enter invalid credentials, **When** I submit the login form, **Then** the system displays a generic "Invalid email or password" error without revealing which field is incorrect.

4. **Given** I have failed login 5 or more times, **When** I attempt another login, **Then** my account is temporarily locked and the system displays a lockout message with contact information.

5. **Given** I am an authenticated user, **When** the system validates my session, **Then** my user role (Patient/Staff/Admin) is determined from the JWT claims and the appropriate dashboard is displayed.

## Edge Cases

* What happens when a deactivated account attempts login? System displays "Account not active. Please contact support." without specifying deactivation.
* How does the system handle concurrent sessions from different devices? Each login creates a new session token; previous sessions coexist until they expire.
