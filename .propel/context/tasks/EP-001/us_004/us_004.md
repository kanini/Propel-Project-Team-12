# User Story - US_004

## Story ID
* ID: US_004

## Story Title
* Automated Preferred Slot Swap Execution

## Description
* As the system, I want to automatically monitor preferred slot availability and execute swaps when slots open, so that patients are moved to their preferred times without manual intervention.

## Acceptance Criteria
1. **Given** there are pending swap requests in the system, **When** the PreferredSlotMonitorJob runs every 5 minutes, **Then** it checks each preferred slot's availability and identifies slots that have opened.

2. **Given** a preferred slot becomes available, **When** the monitoring job detects availability, **Then** it automatically swaps the patient's appointment from the primary slot to the preferred slot, releases the primary slot, and updates the swap request status to "Swapped."

3. **Given** a swap is executed successfully, **When** the transaction completes, **Then** the system sends email and SMS notifications to the patient with the new appointment details and logs the swap action in the audit trail with timestamp and affected appointment IDs.

4. **Given** a swap request is processing, **When** the preferred slot becomes unavailable during the transaction (race condition), **Then** the system aborts the swap, leaves the patient in the primary slot, logs the failed attempt, and retries on the next monitoring cycle.

## Edge Cases
* What happens when multiple patients have swap requests for the same preferred slot? (FIFO processing: first registered swap request wins, others remain pending for future opportunities)
* How does the system handle swaps when the appointment is within 24 hours? (Swap validation: if preferred slot is <24 hours away, system skips swap to avoid disrupting imminent appointments, swap request remains "Pending" until expired)
* What happens if email/SMS notification fails after successful swap? (Swap remains completed; system retries notification 3x with exponential backoff; logs failure if all retries fail; admin alerted for manual outreach)
* How does the system handle calendar sync failures during swap? (Swap completes successfully; system logs calendar sync failure; patient notified to manually update calendar via provided .ics file; retry attempted on next sync cycle)
