# User Story - US_020

## Story ID

* ID: US_020

## Story Title

* Role-Based Access Control Enforcement

## Description

* As a platform administrator, I want role-based access control enforced for Patient, Staff, and Admin roles with minimum necessary access, so that users can only access functionality appropriate to their role.

## Acceptance Criteria

1. **Given** RBAC is implemented, **When** a Patient accesses a Staff-only endpoint (e.g., queue management), **Then** the API returns 403 Forbidden and the frontend does not render Staff navigation items.

2. **Given** route guards are active, **When** a Staff user attempts to access Admin-only pages (e.g., user management), **Then** the frontend redirects to the Staff dashboard and the API rejects the request with 403.

3. **Given** minimum necessary access is enforced (NFR-014), **When** a Patient requests another patient's data, **Then** the API returns 403 and logs the unauthorized access attempt.

4. **Given** role-based navigation is configured, **When** each role logs in, **Then** the sidebar/navigation only displays menu items available to that role: Patient sees appointments/intake/documents/dashboard, Staff sees queue/walk-in/verification/patients, Admin sees users/audit/settings.

## Edge Cases

* What happens when a user's role is changed while they have an active session? Upon next API call, the middleware should re-validate the role from the token and reject if role was downgraded.
* How does the system handle requests with tampered JWT role claims? RS256 signature verification rejects any tampered tokens with 401 Unauthorized.
