# User Story - US_005

## Story ID

- ID: US_005

## Story Title

- Role-Specific Portal Navigation and Styling

## Description

- As an authenticated user, I want to see a portal interface styled and navigated according to my role (Patient, Staff, or Admin), so that I can quickly identify my context and access only the features relevant to my responsibilities.

## Acceptance Criteria

1. **Given** I am logged in as a Patient, **When** the dashboard loads, **Then** I see patient-specific branding (calming blue-green palette), navigation items (Dashboard, Appointments, Intake, Documents, Profile), and cannot access Staff or Admin features.

2. **Given** I am logged in as Staff, **When** the dashboard loads, **Then** I see staff-specific branding (distinct accent color), navigation items (Queue, Patient Search, Walk-in, Arrivals, Insurance, Conflicts, Code Verification), and cannot access Admin features.

3. **Given** I am logged in as Admin, **When** the dashboard loads, **Then** I see admin-specific branding (distinct accent color), navigation items (Dashboard, Users, Roles, Audit Logs), and have access to administrative functions only.

4. **Given** I am authenticated with any role, **When** I view the navigation sidebar, **Then** it displays consistently in the same position across all authenticated screens (collapsible on desktop, bottom nav on mobile).

5. **Given** I am authenticated, **When** I view the header, **Then** I see my name/avatar, current role indicator, and a user menu with settings and logout options.

6. **Given** I am a Patient attempting to access a Staff URL directly (e.g., /staff/queue), **When** the page loads, **Then** the system returns a 403 Forbidden response and redirects to the Patient dashboard with a message "Access denied".

7. **Given** I am viewing the portal on mobile (viewport < 768px), **When** I look for navigation, **Then** I see a bottom navigation bar or hamburger menu appropriate for my role.

## Edge Cases

- What happens if a user's role is changed while logged in? (Role change takes effect on next login; current session maintains original role)
- How does the system handle deep linking to role-specific pages before authentication? (Redirect to login, then to requested page if authorized)
- What happens if role configuration data fails to load? (Show error state with retry option; do not expose unauthorized navigation)
- How does the system handle users with Staff role accessing Patient-shared screens like 360-View? (Staff see SCR-009 in Staff context with staff-specific actions)
