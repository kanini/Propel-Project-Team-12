# User Story - US_034

## Story ID
* ID: US_034

## Story Title
* Admin Deactivate User Account

## Description
* As an Admin, I want to deactivate user accounts to prevent login without deleting historical data, so that I can revoke access for terminated employees or inactive patients while preserving audit trails.

## Acceptance Criteria
1. **Given** I am viewing the user management list, **When** I click the "Deactivate" button for an active user, **Then** the system displays a confirmation dialog with message "Deactivate {userName}? Active sessions will be terminated. This action can be reversed by reactivating the account," a reason text field (optional), and "Confirm" and "Cancel" buttons.

2. **Given** I confirm deactivation, **When** the system calls `POST /api/admin/users/{id}/deactivate` with optional reason, **Then** the backend updates the user's account status to "Inactive," sets deactivatedAt timestamp, invalidates all active sessions for the user by deleting entries from ActiveSessions table, logs the deactivation with admin ID and reason in audit log, and returns 200 OK.

3. **Given** the deactivation is successful, **When** the API responds, **Then** the user list refreshes, the deactivated user's status displays "Inactive" badge in red, action buttons change to show "Reactivate" instead of "Deactivate," and a success toast displays "User deactivated successfully."

4. **Given** a deactivated user attempts to log in, **When** the authentication service validates credentials, **Then** the system checks account status, finds "Inactive," rejects login with 403 Forbidden status, displays error message "Your account has been deactivated. Contact your administrator," and logs the failed login attempt with userID and timestamp.

## Edge Cases
* What happens when I try to deactivate a user with upcoming appointments? (System displays warning "User has 3 upcoming appointments. Deactivate anyway?" with appointment list; admin can proceed or cancel; deactivation does not cancel appointments but user cannot access them)
* How does the system handle reactivating a deactivated user? (Admin clicks "Reactivate" button; system updates status to "Active," logs reactivation, sends optional notification email to user; user can log in immediately with existing password)
* What happens if the deactivated user was logged in on multiple devices? (All sessions across all devices invalidated; user receives 401 Unauthorized on next API request; frontend redirects to login on any tab)
* Can a deactivated Admin user be reactivated? (Yes, another Admin can reactivate; system validates that reactivating admin has sufficient privileges; audit log records both deactivation and reactivation with justification)
