# User Story - US_035

## Story ID
* ID: US_035

## Story Title
* Admin Reset User Password

## Description
* As an Admin, I want to initiate password resets for users via email link, so that users who forget their passwords or need password changes can regain account access securely without admin knowing the new password.

## Acceptance Criteria
1. **Given** I am viewing the user management list, **When** I click the "Reset Password" button for a user, **Then** the system displays a confirmation dialog "Send password reset email to {userEmail}?" with "Confirm" and "Cancel" buttons.

2. **Given** I confirm the password reset, **When** the system calls `POST /api/admin/users/{id}/reset-password`, **Then** the backend generates a unique password reset token (GUID), stores it in the PasswordResetTokens table with userID and expiration timestamp (NOW() + 1 hour), sends a password reset email via SendGrid/Mailgun containing the reset link with token, and returns 200 OK with emailSent status.

3. **Given** the reset email is sent successfully, **When** the API responds, **Then** the system displays a success toast "Password reset email sent to {userEmail}," and logs the reset initiation in the audit log with admin ID, userID, and timestamp.

4. **Given** the user receives the reset email, **When** the user clicks the reset link, **Then** the system validates the token (checks expiration and userID match), displays a password reset form if valid, or shows "Reset link expired or invalid. Request a new reset" error if token is expired or invalid.

## Edge Cases
* What happens when the password reset email bounces (invalid email)? (Email service returns bounce notification; backend logs bounce event; admin sees warning "Email bounced. Generate manual temp password?"; admin can click to generate and display temp password securely)
* How does the system handle multiple password reset requests for the same user? (Each new reset request invalidates previous tokens; only the most recent token is valid; old tokens return "Token already used or invalid" error)
* What happens if the reset link is expired when user clicks it? (Token expiration check fails; user sees error page with "Reset link expired. Request a new reset" message and "Back to Login" button; user must request new reset via "Forgot Password" or admin action)
* How does the system ensure reset emails comply with HIPAA? (Email contains minimal PHI (name only, no medical data); reset link uses opaque token (no userID in URL); TLS 1.2+ enforced for email transmission; audit log records all reset requests)
