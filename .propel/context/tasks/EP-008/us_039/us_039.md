# User Story - US_039

## Story ID
* ID: US_039

## Story Title
* Password Policy Validation

## Description
* As the authentication system, I want to enforce password complexity requirements (min 8 characters, uppercase, lowercase, digit, special character) during registration, password reset, and password change, so that user accounts are protected from weak password attacks.

## Acceptance Criteria
1. **Given** a user enters a new password during account activation, password reset, or password change, **When** the password field loses focus, **Then** the system validates the password against policy requirements (min 8 chars, at least 1 uppercase letter, at least 1 lowercase letter, at least 1 digit, at least 1 special character) and displays real-time validation feedback with green checkmarks for met requirements and red X marks for unmet requirements.

2. **Given** a user submits a password that does not meet policy requirements, **When** the form is submitted to the backend, **Then** the API revalidates the password server-side, rejects the request with 400 Bad Request status, returns error message listing unmet requirements ("Password must contain at least 1 uppercase letter, 1 digit"), and logs the validation failure in the audit log.

3. **Given** a user enters a valid password meeting all policy requirements, **When** the backend processes the password, **Then** the system hashes the password using bcrypt with work factor 12 (2^12 = 4096 iterations), stores the hashed password in the database passwordHash column, and never stores the plaintext password.

4. **Given** a user changes their password, **When** the new password is submitted, **Then** the system validates that the new password is different from the current password (compare hashes), rejects if identical with error "New password must be different from current password," and logs the password change event in the audit log with userID and timestamp (not the password itself).

## Edge Cases
* What happens if a user enters a password with only spaces or special characters? (Validation rejects passwords with only whitespace; special character requirement met by symbols like !@#$%^&*; password trimmed before validation to remove leading/trailing spaces)
* How does the system communicate password requirements before the user starts typing? (Password field displays helper text below input: "Password must be at least 8 characters and include uppercase, lowercase, digit, and special character"; requirements list shown with unchecked boxes initially)
* What happens if a user copies and pastes a password with hidden characters? (Password validation processes all Unicode characters; hidden characters count toward length; paste allowed; validation feedback updates immediately after paste)
* How does the system prevent common passwords (e.g., "Password123!")? (Future enhancement: integrate with common password dictionary (e.g., haveibeenpwned API); Phase 1: focus on complexity requirements only; log common password detection as technical debt)
