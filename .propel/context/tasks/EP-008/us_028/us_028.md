# User Story - US_028

## Story ID
* ID: US_028

## Story Title
* JWT Token Generation and RBAC Infrastructure

## Description
* As the authentication system, I want to generate JWT tokens with role claims upon successful login, so that user role and permissions can be validated on every API request without database lookups.

## Acceptance Criteria
1. **Given** a user successfully authenticates with valid credentials, **When** the authentication service validates the user, **Then** the system generates a JWT token containing claims (`sub` = UserID, `email`, `role` = Patient/Staff/Admin, `grantedPermissions` = array of feature flags) and signs it with the secret key stored in environment variables.

## 2. **Given** a JWT token is generated, **When** the token is issued to the client, **Then** the token expiration is set to 15 minutes (900 seconds), the token includes `iat` (issued at) and `exp` (expiration) timestamps, and the response includes the token in an httpOnly secure cookie.

3. **Given** the API receives a request with a JWT token, **When** the JWT middleware validates the token, **Then** the system verifies the signature using the secret key, checks token expiration, extracts role claims, and rejects expired or invalid tokens with 401 Unauthorized status.

4. **Given** a valid JWT token is validated, **When** the middleware extracts user claims, **Then** the system populates the request context with UserID, email, role, and grantedPermissions, making them accessible to downstream controllers and services for authorization decisions.

## Edge Cases
* What happens when the JWT secret key is rotated? (System supports graceful key rotation: old tokens remain valid until expiration while new tokens use new key; dual-key validation during rotation window)
* How does the system handle JWT tokens with tampered claims? (Signature verification fails, middleware rejects with 401 Unauthorized, logs security event with IP address and attempted claims)
* What happens if the token expiration claim is missing or malformed? (JWT validation library rejects token as invalid, returns 401 Unauthorized, logs validation error)
* How does the system handle clock skew between client and server? (JWT validation allows 60-second clock skew tolerance via `ClockSkew` configuration; tokens within skew window are accepted)
