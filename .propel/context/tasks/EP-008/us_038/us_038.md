# User Story - US_038

## Story ID
* ID: US_038

## Story Title
* Concurrent Session Blocking

## Description
* As the authentication system, I want to prevent users from being logged in on multiple devices simultaneously, so that credential sharing is discouraged and security accountability is maintained.

## Acceptance Criteria
1. **Given** a user logs in successfully, **When** the authentication service validates credentials, **Then** the system generates a unique session ID (GUID), stores it in the ActiveSessions table with columns (sessionID, userID, deviceFingerprint, ipAddress, userAgent, loginTimestamp), and includes the sessionID in the JWT token claims.

2. **Given** a user with an active session logs in from a different device, **When** the new login request is processed, **Then** the system deletes the previous session from the ActiveSessions table (based on userID), generates a new sessionID for the new device, sends the updated JWT token to the new device, and logs the session replacement event in audit log with both old and new sessionIDs.

3. **Given** a user has an active session on Device A, **When** the user logs in on Device B, **Then** the next API request from Device A receives 401 Unauthorized with error "Session invalidated. You have been logged in from another location," the frontend on Device A detects the 401 error, displays logout notification, and redirects to the login page.

4. **Given** an API request contains a JWT token, **When** the RBAC middleware validates the request, **Then** the system extracts the sessionID from the token, queries the ActiveSessions table for a matching sessionID and userID pair, allows the request if found, or rejects with 401 Unauthorized if not found (indicating session was invalidated by concurrent login).

## Edge Cases
* What happens if the same user logs in rapidly from the same device (e.g., multiple browser tabs)? (Each login generates a new sessionID; previous session invalidated; all tabs except the last login tab are logged out; concurrent session blocking enforced even within single device)
* How does the system differentiate between device changes vs. browser sessions? (DeviceFingerprint generated client-side using FingerprintJS library capturing screen resolution, timezone, plugins, canvas rendering; fingerprint stored with session but not used for validation to allow legitimate browser upgrades)
* What happens if the ActiveSessions table becomes unavailable during session validation? (Database circuit breaker opens; system falls back to JWT token validation only (trusts token expiration); logs fallback event; sessions not invalidated until database recovers)
* How does the system handle logout cleanup? (On logout, DELETE FROM ActiveSessions WHERE sessionID={currentSession}; if user closes browser without logout, session remains in table until next login or TTL-based cleanup job runs)
