# User Story - US_040

## Story ID
* ID: US_040

## Story Title
* User Login with RBAC

## Description
* As a user with any role (Patient/Staff/Admin), I want to log in with my email and password to receive a role-appropriate JWT token, so that I can access features permitted by my assigned role throughout the application.

## Acceptance Criteria
1. **Given** I am on the login screen, **When** I enter my email and password and click "Log In," **Then** the system calls `POST /api/auth/login` with credentials, validates that the account is active (accountStatus='Active'), verifies the password hash matches using bcrypt comparison, generates a JWT token with my role claims, and returns 200 OK with the token in an httpOnly secure cookie.

2. **Given** I successfully log in as a Patient, **When** the JWT token is issued, **Then** the token includes claims (`sub` = UserID, `email`, `role` = Patient, `grantedPermissions` = [BookAppointments, CompleteIntake, UploadDocuments]), the frontend stores my role in React Context, and I am redirected to the Patient Dashboard (SCR-001) with navigation showing only Patient menu items.

3. **Given** I successfully log in as Staff or Admin, **When** the JWT token is issued, **Then** the token includes role-specific claims (Staff: WalkIns, ClinicalReview, Arrivals; Admin: UserManagement, AuditLogs, SystemConfig), the frontend redirects me to my role-appropriate dashboard, and navigation displays only my permitted menu items based on role.

4. **Given** I enter invalid credentials, **When** the authentication service validates the credentials, **Then** the system returns 401 Unauthorized with generic error message "Invalid email or password" (does not reveal whether email exists), logs the failed login attempt with email, IP address, and timestamp in audit log, and implements rate limiting to prevent brute force attacks (max 5 failed attempts per 15 minutes per IP before temporary lockout).

## Edge Cases
* What happens when I try to log in with a deactivated account? (System checks accountStatus before password validation; returns 403 Forbidden with error "Your account has been deactivated. Contact your administrator"; does not reveal password correctness; logs failed login attempt)
* How does the system handle case-insensitive email login? (Email comparison uses case-insensitive matching; `user@EXAMPLE.COM` matches `user@example.com`; database column uses CITEXT type or LOWER() function in query)
* What happens if I'm already logged in on another device and log in again? (Concurrent session blocking (US-038) invalidates previous session; new login generates new sessionID; old device logged out on next API request)
* How does the system prevent timing attacks on password validation? (Bcrypt comparison executes in constant time regardless of match/mismatch; failed login response delay randomized between 200-500ms to obscure validation timing differences)

## Effort Estimation
* **Story Points**: 4
* **Estimated Hours**: 32
* **Complexity**: High
* **Rationale**: Implements POST /api/auth/login endpoint with credential validation, bcrypt password hash comparison, account status checking (active vs inactive), JWT token generation with role and permission claims, httpOnly secure cookie response, login form UI with email/password fields and validation, role-based redirect logic to appropriate dashboard, failed login attempt logging with rate limiting (5 attempts per 15 minutes), timing attack mitigation via constant-time comparison and response delays, and audit logging for all login events. Complex due to multi-factor validation, security hardening (rate limiting, timing attacks), and role-based post-login routing.
