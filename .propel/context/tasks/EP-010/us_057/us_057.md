# User Story - US_057

## Story ID

* ID: US_057

## Story Title

* Access Control & API Monitoring

## Description

* As a Security Engineer, I want API endpoints protected by role-based access control with rate limiting and real-time monitoring, so that unauthorized access is prevented and abuse is detected quickly.

## Acceptance Criteria

1. **Given** the RBAC requirement (FR-043), **When** a request arrives at a protected endpoint, **Then** the JWT middleware validates the token, extracts the role claim, and rejects requests with 403 Forbidden if the user's role lacks the required permission.

2. **Given** rate limiting (NFR-011), **When** a client exceeds the rate limit threshold, **Then** the API returns 429 Too Many Requests with a Retry-After header, and the event is logged to the audit system.

3. **Given** API monitoring, **When** requests are processed, **Then** response times, error rates, and request volumes are tracked and available via a health/metrics endpoint within 1 second of query.

4. **Given** suspicious activity, **When** repeated 401/403 errors occur from the same IP (>10 in 5 minutes), **Then** the system logs a security alert and optionally triggers temporary IP throttling.

## Edge Cases

* What happens when the JWT signing key is rotated? The system supports key rotation with a grace period where both old and new keys are accepted for 24 hours.
* How are public endpoints (login, registration) protected without JWT? Rate limiting and CAPTCHA verification apply to unauthenticated endpoints.
