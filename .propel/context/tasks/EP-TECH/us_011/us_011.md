# User Story - US_011

## Story ID
* ID: US_011

## Story Title
* Health Check Endpoints

## Description
* As a DevOps engineer, I want `/health/live` and `/health/ready` endpoints that report API liveness and database connectivity, so that monitoring tools and orchestrators can detect and restart unhealthy instances automatically.

## Acceptance Criteria
1. **Given** the .NET 8 API is running, **When** I send a GET request to `/health/live`, **Then** I receive:
   - HTTP 200 OK response if the API process is running
   - JSON response body: `{"status": "Healthy", "checks": []}`
   - Response time <100ms

2. **Given** the .NET 8 API is running and database is connected, **When** I send a GET request to `/health/ready`, **Then** I receive:
   - HTTP 200 OK response if API is ready to serve requests (database connection successful)
   - JSON response body: `{"status": "Healthy", "checks": [{"name": "database", "status": "Healthy"}]}`
   - Response time <500ms

3. **Given** the database is unavailable or connection fails, **When** I send a GET request to `/health/ready`, **Then** I receive:
   - HTTP 503 Service Unavailable response
   - JSON response body: `{"status": "Unhealthy", "checks": [{"name": "database", "status": "Unhealthy", "description": "Connection failed"}]}`

4. **Given** health check endpoints are documented, **When** I read the API documentation or README, **Then** I find:
   - `/health/live` - Liveness probe for orchestrator restart signals (no external dependencies checked)
   - `/health/ready` - Readiness probe with database connectivity check
   - Recommended health check intervals and timeout settings

## Edge Cases
* What happens when health check endpoints are overwhelmed by monitoring requests? (Configure rate limiting or response caching with 5-second TTL)
* How does the system handle transient database connection failures during health checks? (Implement retry with exponential backoff for database check, fail after 3 attempts)

## Technical Notes
- Install health check package:
  ```bash
  dotnet add package Microsoft.Extensions.Diagnostics.HealthChecks.EntityFrameworkCore
  ```
- Configure health checks in `Program.cs`:
  ```csharp
  using Microsoft.Extensions.Diagnostics.HealthChecks;

  var builder = WebApplication.CreateBuilder(args);

  // Add health checks
  builder.Services.AddHealthChecks()
      .AddDbContextCheck<ApplicationDbContext>(
          name: "database",
          failureStatus: HealthStatus.Unhealthy,
          tags: new[] { "ready" });

  var app = builder.Build();

  // Liveness probe - no external dependencies
  app.MapHealthChecks("/health/live", new HealthCheckOptions
  {
      Predicate = _ => false, // Exclude all checks
      ResponseWriter = async (context, report) =>
      {
          context.Response.ContentType = "application/json";
          var result = System.Text.Json.JsonSerializer.Serialize(new
          {
              status = "Healthy",
              checks = Array.Empty<object>()
          });
          await context.Response.WriteAsync(result);
      }
  });

  // Readiness probe - includes database check
  app.MapHealthChecks("/health/ready", new HealthCheckOptions
  {
      Predicate = check => check.Tags.Contains("ready"),
      ResponseWriter = async (context, report) =>
      {
          context.Response.ContentType = "application/json";
          var result = System.Text.Json.JsonSerializer.Serialize(new
          {
              status = report.Status.ToString(),
              checks = report.Entries.Select(e => new
              {
                  name = e.Key,
                  status = e.Value.Status.ToString(),
                  description = e.Value.Description
              })
          });
          await context.Response.WriteAsync(result);
      }
  });

  app.Run();
  ```
- Liveness check: Simple HTTP 200 response proving API process is running
- Readiness check: Executes `DbContext.Database.CanConnectAsync()` to verify database connectivity
- Document health check endpoints in OpenAPI/Swagger with example responses
