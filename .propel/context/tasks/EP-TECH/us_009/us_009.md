# User Story - US_009

## Story ID
* ID: US_009

## Story Title
* CI/CD Pipeline - Lint & Security Scanning

## Description
* As a DevOps engineer, I want GitHub Actions workflows that enforce code linting and security scanning for both backend and frontend, so that code style violations and known vulnerabilities are caught before merge.

## Acceptance Criteria
1. **Given** the GitHub Actions workflows exist, **When** I inspect `.github/workflows/`, **Then** I find lint and security steps integrated into existing CI workflows or separate workflow files.

2. **Given** the frontend linting is configured, **When** code is pushed, **Then** GitHub Actions executes:
   - `npm run lint` (ESLint) to check TypeScript/React code style
   - Workflow fails if ESLint reports errors (warnings allowed)

3. **Given** the security scanning is configured, **When** code is pushed, **Then** GitHub Actions executes:
   - `npm audit --production` to check for known vulnerabilities in frontend dependencies
   - Workflow fails if high or critical severity vulnerabilities are found

4. **Given** SonarCloud is integrated (optional but recommended), **When** a pull request is created, **Then** SonarCloud analyzes code quality and security vulnerabilities for both backend (.NET) and frontend (TypeScript/React), and displays results as PR comment.

## Edge Cases
* What happens when ESLint configuration conflicts with Prettier formatting? (Configure ESLint to extend Prettier rules and disable conflicting rules)
* How does npm audit handle false positive vulnerabilities? (Document exception process using `.npmrc` or `audit-ci` tool with allowed vulnerability list)

## Technical Notes
- Add linting step to `.github/workflows/frontend-ci.yml`:
  ```yaml
  - name: Lint code
    run: npm run lint
    working-directory: ./client
  ```
- Add security scanning step to `.github/workflows/frontend-ci.yml`:
  ```yaml
  - name: Security audit
    run: npm audit --production --audit-level=high
    working-directory: ./client
  ```
- Configure ESLint in `/client/.eslintrc.json`:
  ```json
  {
    "extends": [
      "next/core-web-vitals",
      "plugin:@typescript-eslint/recommended",
      "prettier"
    ],
    "rules": {
      "@typescript-eslint/no-unused-vars": "error",
      "@typescript-eslint/no-explicit-any": "warn"
    }
  }
  ```
- Add lint script to `/client/package.json`:
  ```json
  {
    "scripts": {
      "lint": "next lint",
      "lint:fix": "next lint --fix"
    }
  }
  ```
- SonarCloud integration (optional):
  1. Create SonarCloud account and project
  2. Add `SONAR_TOKEN` secret to GitHub repository settings
  3. Create `sonar-project.properties` in project root:
     ```properties
     sonar.organization=your-org
     sonar.projectKey=patient-access-platform
     sonar.sources=client/app,client/components,api/Domain,api/Application
     sonar.tests=client/__tests__,api/API.Tests
     sonar.javascript.lcov.reportPaths=client/coverage/lcov.info
     sonar.cs.opencover.reportsPaths=api/coverage/**/coverage.opencover.xml
     ```
  4. Add SonarCloud action to workflow:
     ```yaml
     - name: SonarCloud Scan
       uses: SonarSource/sonarcloud-github-action@master
       env:
         GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
         SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
     ```
