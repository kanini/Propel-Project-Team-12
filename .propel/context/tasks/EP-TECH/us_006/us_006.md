# User Story - US_006

## Story ID

* ID: US_006

## Story Title

* Session Caching with Upstash Redis

## Description

* As a developer, I want Upstash Redis configured for session token storage with 15-minute TTL, so that session management is fast, secure, and automatically expires inactive sessions.

## Acceptance Criteria

1. **Given** Upstash Redis is the cache provider, **When** the application starts, **Then** a Redis connection is established using TLS with connection string from environment configuration.

2. **Given** session tokens are stored in Redis, **When** a user authenticates, **Then** the session token is stored with a 15-minute TTL that automatically evicts expired sessions.

3. **Given** the Zero-PHI caching strategy (AD-005), **When** I inspect cached data, **Then** only session tokens and non-PHI metadata are stored — no patient health information exists in Redis.

4. **Given** Redis is operational, **When** a session token is validated, **Then** the lookup completes within 10ms and the TTL is refreshed on valid access (sliding expiration).

## Edge Cases

* What happens when Redis is temporarily unavailable? System should fall back to database-based session validation with degraded but functional operation.
* How does the system handle Redis connection pool exhaustion? Connection should be managed with a pooled client and retry with exponential backoff.
