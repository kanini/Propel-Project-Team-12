# User Story - US_007

## Story ID

* ID: US_007

## Story Title

* CI/CD Pipeline and Free-Tier Deployment

## Description

* As a development team, I want GitHub Actions CI/CD pipelines deploying to free-tier platforms (Vercel, Railway/Render, Supabase) with Windows Services/IIS support, so that automated builds and deployments are reliable and cost-free.

## Acceptance Criteria

1. **Given** GitHub Actions is the CI platform, **When** a pull request is opened, **Then** a CI workflow runs build, lint, and test steps for both frontend and backend with pass/fail status reported on the PR.

2. **Given** Vercel is the frontend host, **When** the CI pipeline succeeds on the main branch, **Then** the React frontend is automatically deployed to Vercel with environment-specific configuration.

3. **Given** Railway/Render is the backend host, **When** the CI pipeline succeeds on the main branch, **Then** the .NET 8 backend is built, containerized, and deployed to the configured platform.

4. **Given** Windows deployment is required, **When** the backend is published using `dotnet publish`, **Then** a self-contained Windows executable is produced that can run as a Windows Service or under IIS.

5. **Given** deployment targets are configured, **When** I inspect the GitHub Actions workflows, **Then** separate workflows exist for CI (`ci.yml`), staging deployment (`cd-staging.yml`), and production deployment (`cd-production.yml`).

## Edge Cases

* What happens when a deployment to Vercel fails due to free-tier limits? Workflow should fail with clear error and notify via GitHub status check.
* How does the system handle concurrent deployments? Deployment workflows should use concurrency groups to prevent simultaneous deploys.
