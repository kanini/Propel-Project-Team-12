# User Story - US_004

## Story ID
   * ID Format: US_004

## Story Title
   * Redis Cache Configuration for Session Management

## Description
  * As a developer, I want to configure Upstash Redis for session token storage with 15-minute TTL, so that user sessions are managed securely with automatic expiration and high-performance access without storing PHI in the cache layer.

## Acceptance Criteria
  * **Given** I have an Upstash account, **When** I create a Redis database, **Then** I receive a connection string with TLS support and REST API endpoint for serverless access.

  * **Given** the backend is configured with StackExchange.Redis, **When** the application starts, **Then** it successfully connects to Upstash Redis using the connection string with TLS enabled.

  * **Given** session service is implemented, **When** a user logs in and receives a JWT token, **Then** the session token is stored in Redis with a 15-minute TTL matching NFR-005 requirements.

  * **Given** a session token exists in Redis, **When** the user makes an API request within 15 minutes, **Then** the session is validated successfully and the TTL is refreshed for the next 15 minutes.

  * **Given** a session has expired, **When** the user makes an API request after 15 minutes of inactivity, **Then** the request is rejected with 401 Unauthorized and the frontend redirects to login.

  * **Given** the cache configuration is set, **When** I verify cache contents, **Then** only session tokens and non-PHI data are stored; no Protected Health Information is cached per AD-005 zero-PHI caching strategy.

## Edge Cases
   * What happens when Redis connection fails? (Application continues with in-memory fallback cache; log warning and alert monitoring)
   * How does the system handle Redis memory eviction? (Configure LRU eviction policy; session tokens have explicit TTL so no premature eviction)
   * What if TTL cannot be set for a key? (Log error, reject session creation, and return service unavailable to user)
