# User Story - US_013

## Story ID
* ID: US_013

## Story Title
* Database Migration Tooling

## Description
* As a backend developer, I want EF Core migration commands and zero-downtime migration strategy documented, so that schema changes can be applied safely to production without service interruption.

## Acceptance Criteria
1. **Given** EF Core is configured, **When** I run `dotnet ef --version` in the `/api` folder, **Then** the command displays EF Core CLI tools version 8.x.

2. **Given** schema changes are required, **When** I run `dotnet ef migrations add <MigrationName>` in the `/api` folder, **Then** a new migration file is generated in `/Infrastructure/Data/Migrations` with:
   - `Up()` method containing schema changes (CREATE, ALTER, DROP)
   - `Down()` method containing rollback operations
   - Migration file named with timestamp prefix (e.g., `20240319_CreatePatientTable.cs`)

3. **Given** a migration is created, **When** I run `dotnet ef database update`, **Then** the migration is applied to the database and:
   - `__EFMigrationsHistory` table records the migration execution timestamp
   - Schema changes are reflected in the database (verified by inspecting tables)
   - Command completes without errors

4. **Given** zero-downtime migration strategy is documented, **When** I read the migration guide (e.g., `MIGRATIONS.md`), **Then** I find:
   - **Expand phase**: Add new columns/tables without removing old schema (backward compatible)
   - **Migrate phase**: Deploy application version that supports both old and new schema
   - **Contract phase**: Remove deprecated columns/tables after application cutover
   - Rollback procedure for failed migrations
   - Example migration scenarios (adding column, renaming table, changing data type)

## Edge Cases
* What happens when a migration fails mid-execution? (Document manual rollback using `dotnet ef migrations remove` and database restore procedures)
* How does the system handle migrations in multi-developer environments with concurrent schema changes? (Document migration branching strategy and merge conflict resolution)

## Technical Notes
- EF Core CLI commands reference:
  ```bash
  # Create new migration
  dotnet ef migrations add <MigrationName> --project ./Infrastructure --startup-project ./API

  # Apply migrations to database
  dotnet ef database update --project ./Infrastructure --startup-project ./API

  # Revert last migration (if not applied to database)
  dotnet ef migrations remove --project ./Infrastructure --startup-project ./API

  # Generate SQL script from migrations
  dotnet ef migrations script --idempotent --output migration.sql --project ./Infrastructure --startup-project ./API

  # List all migrations
  dotnet ef migrations list --project ./Infrastructure --startup-project ./API
  ```
- Zero-downtime migration pattern (Expand-Migrate-Contract):
  1. **Expand**: Add new column `Email_New`, keep old column `Email`
  2. **Migrate**: Deploy app version supporting both `Email` and `Email_New`
  3. **Contract**: Remove old column `Email` after data migration complete
- Create `MIGRATIONS.md` with:
  - Migration command reference
  - Zero-downtime migration strategy
  - Rollback procedures
  - Example scenarios (add column, rename table, change data type)
  - Production deployment checklist
- Configure EF Core to apply migrations on application startup (Development only):
  ```csharp
  if (app.Environment.IsDevelopment())
  {
      using var scope = app.Services.CreateScope();
      var dbContext = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
      dbContext.Database.Migrate(); // Auto-apply pending migrations
  }
  ```
- Production migrations should use SQL scripts generated by `dotnet ef migrations script` and executed by DBA
