# User Story - US_072

## Story ID

* ID: US_072

## Story Title

* Conflict Resolution Workflow for Critical Data Conflicts

## Description

* As a staff member, I want a dedicated Conflict Resolution page (SCR-024) to review and resolve critical data conflicts detected by the aggregation service (e.g., conflicting medications, inconsistent diagnoses), so that I can make informed clinical decisions and ensure patient safety by resolving conflicting information from multiple source documents.

## Acceptance Criteria

1. **Given** critical conflicts are detected during data aggregation, **When** I view Staff Patient View (SCR-017), **Then** the system displays a conflict alerts banner at the top of the page showing conflict count, highest severity level (Critical/Warning), and "Resolve Conflicts" button that navigates to Conflict Resolution page (SCR-024).

2. **Given** I access Conflict Resolution page, **When** the page loads, **Then** the system displays all unresolved conflicts for the patient grouped by conflict type (Medication Conflict, Diagnosis Conflict, Allergy Conflict, Vital Conflict) with each conflict showing: severity badge (red for Critical, amber for Warning), conflicting values side-by-side, source document references with page numbers, extraction confidence scores, and conflict detection reason.

3. **Given** I am resolving a conflict, **When** I select a resolution action, **Then** I can choose: "Select Value A" (accept first value, reject second), "Select Value B" (accept second value, reject first), "Accept Both" (mark both as valid if clinically appropriate), "Enter Custom Value" (opens inline editor for staff-entered resolution), or "Flag for Review" (escalate to senior clinical staff), and each action requires a mandatory resolution note explaining the decision.

4. **Given** I have resolved all conflicts, **When** I click "Save Resolutions" button, **Then** the system persists resolution decisions, updates patient profile with resolved values, removes conflict alerts from Staff Patient View, logs full audit trail with staff ID, timestamp, conflict details, resolution action, and notes (NFR-007, AIR-S04), and displays success message "All conflicts resolved successfully".

## Edge Cases

* What happens when a conflict involves more than 2 conflicting values (3+ documents with different values)? Display all conflicting values in a multi-column comparison table with radio button selection for each value plus "Custom Value" option.
* What happens when staff saves resolution but API call fails? Display error banner "Unable to save conflict resolution. Please try again." with retry button, preserve resolution selections in UI state to prevent data loss, and log failed attempt.
* What happens when another staff member resolves the same conflict simultaneously? Use optimistic locking; if conflict detected, display warning "This conflict was already resolved by [Staff Name] at [Time]. Refreshing page..." and auto-refresh after 2 seconds.
* What happens when new conflicting data arrives while staff is resolving conflicts? Pusher real-time notification displays banner "New conflicting data detected. Please refresh to see updated conflicts." with refresh button.
