# User Story - US_066

## Story ID

* ID: US_066

## Story Title

* Error Handling Patterns

## Description

* As a Patient, I want clear, actionable error messages when something goes wrong, with consistent error presentation across the application, so that I understand the problem and know how to resolve it.

## Acceptance Criteria

1. **Given** inline validation errors (UXR-601), **When** I submit a form with invalid data, **Then** error messages appear inline below the offending field, are styled in red with an error icon, and provide specific guidance (e.g., "Phone number must be 10 digits" not "Invalid input").

2. **Given** API/system errors (UXR-603), **When** a server error occurs, **Then** a toast notification displays with a user-friendly message, an error code for support reference, and a retry action when applicable — no raw error codes or stack traces are shown.

3. **Given** 404 and empty states, **When** a page is not found or a list has no results, **Then** a friendly illustration with a clear message and actionable link (e.g., "Go back to dashboard" or "Create your first appointment") is displayed.

4. **Given** network errors, **When** the user loses connectivity, **Then** a persistent banner announces "You're offline — changes will sync when you reconnect" and the app queues actions for retry.

## Edge Cases

* What happens when multiple validation errors occur simultaneously? All errors display at once with a summary at the top of the form ("3 issues found") and individual inline messages per field; focus shifts to the first error.
* How are errors in modals handled? Modal errors display within the modal context; the modal does not auto-close on error, allowing the user to correct and retry.
