# User Story - US_002

## Story ID

- ID: US_002

## Story Title

- Multi-Role User Login Authentication

## Description

- As a registered user (Patient, Staff, or Admin), I want to log in with my credentials and be directed to my role-appropriate portal, so that I can access features relevant to my responsibilities.

## Acceptance Criteria

1. **Given** I am on the sign-in page with valid credentials, **When** I enter my email and password and click "Sign In", **Then** the system authenticates me and redirects me to my role-specific dashboard (Patient Dashboard, Staff Dashboard, or Admin Dashboard) within 2 seconds.

2. **Given** I am logging in as a Patient, **When** authentication succeeds, **Then** I am redirected to the Patient Portal with patient-specific navigation, color accents, and features (appointments, intake, documents).

3. **Given** I am logging in as Staff, **When** authentication succeeds, **Then** I am redirected to the Staff Portal with staff-specific navigation, color accents, and features (queue management, patient search, code verification).

4. **Given** I am logging in as Admin, **When** authentication succeeds, **Then** I am redirected to the Admin Portal with admin-specific navigation, color accents, and features (user management, roles, audit logs).

5. **Given** I enter incorrect credentials, **When** I click "Sign In", **Then** the system displays "Invalid email or password" without revealing which field is incorrect (security best practice).

6. **Given** my account is not yet verified, **When** I attempt to log in, **Then** the system displays "Please verify your email to continue" with a link to resend the verification email.

7. **Given** I am authenticated, **When** a session JWT token is issued, **Then** the token contains role claims (Patient/Staff/Admin) for authorization throughout the session.

## Edge Cases

- What happens when a user has multiple roles? (System uses primary role for initial redirect; role switching available in header if applicable in future phases)
- How does the system handle login attempts with case variations in email? (Email matching is case-insensitive)
- What happens if the user's account is deactivated? (System displays "Account is inactive. Please contact support.")
- How does the system handle simultaneous logins from multiple devices? (Both sessions are valid; each device receives its own token)
